Mail Digest
v0.4 · 25 Mai 2026 Estado & Roadmap
Estado · Maio 2026

Um briefing matinal do teu email, não outro cliente.

O Mail Digest lê os teus emails de manhã, decide o que importa, e entrega-te um documento curto com o que precisa de ti, o que ele pode tratar, e o que pode ser ignorado. Roda na tua máquina, com o modelo de AI à tua escolha.

Princípio fundador

O email é uma inbox de obrigações escrita por outras pessoas. O Mail Digest transforma-a numa checklist de decisões escrita para ti — mais curta, ordenada por prioridade, e com acções inline. Não somos um cliente de email. Somos um assistente que lê email.

01 · agora

O que está implementado

v0.4 · funcional

App Electron cross-platform (macOS · Windows · Linux) que se liga ao Gmail, lê as últimas 24h, e produz um digest estruturado com um modelo premium à escolha (Claude, GPT, ou Gemini). Corre no teu desktop — não há servidor Mail Digest no meio; os pedidos vão directos para o provider que configuraste.

Digest diário estruturado

Feito

Para cada email é extraído um objecto JSON com prioridade (urgent · today · this_week · fyi), resumo em 1–2 frases, tarefas derivadas, eventos de calendário detectados, e rascunho de resposta quando aplicável. O resultado é ordenado por prioridade e renderizado no painel principal.

Output schema · items[].{priority, summary, needs_reply, tasks[], calendar_events[], draft_reply}

3 providers premium à escolha

Feito

O utilizador escolhe Claude Sonnet, GPT-4.1 Mini, ou Gemini 2.5 Flash nas definições e cola a sua API key. O dropdown alterna provider e modelo sem mudar o workflow. Paga o provider directamente — a app é gratuita (MIT).

@anthropic-ai/sdk openai @google/generative-ai

Provider abstraction

Feito

Toda a parte de fetching de email passa por uma camada de provider. Hoje só existe o Gmail, mas a interface é genérica: fetchRecentMessages(auth, opts) → Message[]. Adicionar Outlook ou IMAP no futuro é só implementar mais um module.

02 · próximo

Iterações com prompts prontos

4 features · ITERATIONS.md

Os prompts detalhados destas quatro features estão escritos em ITERATIONS.md e são executáveis directamente em Cursor / Claude Code. Estimativa total: ~14–17h de trabalho. Aumentam o sinal/ruído do digest e desbloqueiam o conceito de assistente em vez de renderer.

1 · Auto-categorização (important · newsletter · transactional)

Próximo

Adiciona um campo category ao schema do digest. UI passa a três secções: PARA TI, NEWSLETTERS & FEEDS, TRANSACCIONAL. Resolve a sensação actual de "tudo misturado" e prepara o terreno para o passo seguinte.

Estimativa · ~1h · zero deps novas

2 · Compressão de newsletters num único parágrafo

Próximo

Two-pass LLM: classifica primeiro, depois envia o subset de newsletters para um NEWSLETTER_PROMPT dedicado que sintetiza tudo num único parágrafo coeso. Em vez de 8 cards de newsletter, ficas com 1 parágrafo + breakdown colapsável por fonte.

Estimativa · ~3–4h · adiciona newsletter_synthesis ao schema

3 · Detecção de follow-ups (emails que enviaste sem resposta)

Próximo

Vai à pasta Sent, identifica threads onde foste a última a falar, e mostra-os no topo do digest como secção FOLLOW-UPS. Cada item vem com rascunho de re-ping educado pré-gerado. Cap a 10 por digest. Persiste em SQLite (better-sqlite3) para evitar re-pingar items já tratados.

Estimativa · ~6–8h · introduz better-sqlite3 · primeira feature stateful

4 · Reply queue persistente

Próximo

Quando o digest diz needs_reply: true, o item entra numa fila de respostas pendentes. Sai automaticamente quando o Mail Digest detecta um email teu na thread. Resolve o problema do "lembrei-me que tinha de responder àquele email há 3 dias".

Estimativa · ~3–4h · partilha SQLite com follow-ups
03 · depois

Features que fazem o produto diferenciar

5 features · investigação necessária

Estas são as features que justificam a existência do produto face a Superhuman, Shortwave, ou um simples "Gmail + Claude API". Cada uma precisa de algum design antes de ter um prompt executável.

Voice fingerprint — rascunhos que parecem teus

Roadmap

Lê os últimos 200 emails enviados, extrai padrões da tua voz (saudação típica, despedida, tamanho médio de frase, formalidade, uso de markdown, idioma por interlocutor), e usa esse fingerprint em todos os rascunhos sugeridos. Diferenciador real face a draft replies "robóticas" que cheiram a AI à distância.

Inspiração · Lex.page voice match · Superhuman split-test · OpenAI fine-tune local

Push 1-click para Calendar · Reminders · Todoist · Things

Roadmap

O digest já extrai eventos e tarefas. Falta um botão que os cria efectivamente nas apps que usas. Integrações iniciais sugeridas: Google Calendar (API), Apple Reminders (AppleScript em macOS), Todoist (REST API), Things 3 (URL scheme). Transforma o produto de "ferramenta de leitura" em "ferramenta de execução".

Risco · mais APIs externas a gerir · keychain mais cheia

Pattern detection multi-dia

Roadmap

"O João escreveu-te 3 vezes esta semana sobre o projecto X — talvez seja altura de ter uma chamada." "O banco enviou-te alertas de saldo baixo 4x este mês." Padrões só visíveis com histórico. Precisa da camada SQLite (já introduzida na iteração 3) para guardar metadados de emails passados.

Inspiração · Hey Imbox screener · Superhuman insights

Audio briefing — ouve o teu digest a caminho do trabalho

Talvez

Transforma o briefing em ficheiro áudio (TTS local com Piper, ou ElevenLabs / OpenAI TTS opcionalmente). 3–5 minutos. Tocas no comboio, no carro, durante o pequeno-almoço. Posiciona o produto longe de qualquer cliente de email.

Stack · Piper TTS local · OpenAI tts-1 · ElevenLabs

Cost transparency dashboard

Roadmap

Para utilizadores com API keys configuradas: estimativa em tempo real do custo do digest deste mês, partido por tokens de input, tokens de output, e provider. Combate a opacidade que afasta utilizadores de produtos AI.

04 · talvez

Hipóteses mais distantes

5 ideias · valida antes

Ideias que parecem boas mas precisam de utilizadores reais primeiro. Listo aqui para não as perder, mas só ganham prioridade se o feedback do roadmap acima as pedir.

Outlook + IMAP — multi-provider de email

Talvez

A arquitectura já está preparada (provider abstraction). Falta implementar o module Outlook (Microsoft Graph API) e o module IMAP genérico para Fastmail, ProtonMail, etc. Decisão depende de quantos utilizadores reais pedem.

Multi-account no mesmo digest

Talvez

Conectar pessoal + trabalho na mesma app, com um único digest unificado ou views separadas. Útil para freelancers e founders. Complica o modelo OAuth e a UX de definições.

Mobile companion (read-only)

Talvez

App nativa iOS / Android que lê o digest gerado no desktop. Sync via iCloud Drive, Dropbox, ou um endpoint encriptado. Não faz inference — só apresentação.

Modo conversação livre com o assistente

Talvez

"Mostra-me todos os emails do banco nos últimos 30 dias." "Quais são as 3 threads mais importantes que ignorei?" Requer indexação local pesquisável (SQLite FTS5) e um agente que sabe usar essa indexação.

Encriptação at-rest das API keys

Tarefa técnica

Hoje as API keys são guardadas no electron-store sem encriptação adicional (assumimos que o user account está protegido). Mover para AES-256 com chave derivada do password do OS keychain. Pré-requisito para distribuir builds que façam shadowing de keys.

05 · explícito

O que não vamos fazer

non-goals

A clareza sobre o que não é o Mail Digest é tão importante como o roadmap. Estas são decisões deliberadas — não esquecimentos.

×
Não vamos ser um cliente de email completo.
Não fazemos compose-de-zero, gestão de labels, search avançado, ou rules. O Gmail e clientes nativos já fazem isso. Continuamos a ser uma camada por cima.
×
Não corremos servidor próprio que vê os teus emails.
A escolha do provider é tua. O email vai do Gmail para o teu desktop e depois directamente para Claude, GPT, ou Gemini com a tua API key — nunca por um servidor Mail Digest.
×
Não fazemos analytics nem telemetria.
Não sabemos quem usa, quanto usa, ou que features são populares. Custa-nos dados de produto. Ganhamos confiança dos utilizadores. É um trade-off explícito.
×
Não vamos fazer um modelo de subscrição inicial.
Enquanto for MIT + bring-your-own-API-key, o modelo é open-source. Se um dia oferecermos uma versão gerida (com inference partilhada e sync entre máquinas), aí justifica subscrição. Não antes.
×
Não fazemos auto-reply.
Rascunhos sim, envio automático nunca. O custo de uma mensagem mal-enviada em nome do utilizador é alto demais. O humano fica sempre no loop final.
